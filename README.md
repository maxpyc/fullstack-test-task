## 🛠 Отчет о проведенном рефакторинге и оптимизации бэкенда

В ходе работы над проектом был проведен рефакторинг кодовой базы бэкенда, 
исправлены архитектурные ошибки, устранены проблемы с безопасностью типов и 
проведен ряд критически важных оптимизаций для работы приложения под высокой нагрузкой.

---

### 🏛 1. Архитектура и Чистый код (Clean Architecture)

Исходный код имел сильную связность (tight coupling): бизнес-логика и SQL-запросы были перемешаны в слое роутеров, 
использовались глобальные объекты сессий, что делало невозможным модульное тестирование.

**Что сделано:**
* **Разделение на слои:** Код строго разделен по принципу **Presentation (Routers) ➔ Service Layer (Бизнес-логика) ➔ Repository Layer (Доступ к данным)**.
* **Внедрение зависимостей (Dependency Injection):** Избавились от глобальных сессий БД. Теперь сессии и сервисы внедряются в эндпоинты через систему `Depends()` в FastAPI. Это делает систему гибкой и готовой к тестированию.
* **Слой репозиториев (Repository Pattern):** Написан расширяемый `BaseRepository` на основе Generic-типов. Вся работа с SQLAlchemy (`select`, `add`, `commit`, `delete`) инкапсулирована внутри классов `FileRepository` и `AlertRepository`.
* **Доменные исключения:** Бизнес-логика больше не знает про HTTP-коды ответов. Сервисы выбрасывают стандартные исключения Python (`FileNotFoundError`, `ValueError`), а тонкий слой роутеров перехватывает их и маппит в соответствующие `HTTPException`.

---

### 💾 2. Оптимизация Базы Данных (PostgreSQL + SQLAlchemy 2.0)

**Что сделано:**
* **Нативный UUID:** Первичные ключи файлов и внешние ключи алертов переведены со строк `String(36)` на **нативный тип `Uuid`** (в SQLAlchemy 2.0 и PostgreSQL). Это снизило потребление памяти в БД на запись ключа с 37+ байт до **16 байт** и в разы ускорило индексацию и поиск по ID.
* **Типизация:** Схемы переведены на строгий синтаксис **Pydantic V2** с полной поддержкой валидации UUID на входе и автоматической сериализацией.

---

### ⚡ 3. Критические оптимизации I/O и памяти (Решение "неочевидной" задачи)

В исходном коде присутствовали две уязвимости, способные вызвать падение сервера по нехватке памяти (**Out of Memory / OOM**) и зависание из-за блокировки основного потока (Event Loop).

#### А. Потоковое сохранение файлов (Чанкирование)
* **Было:** `await upload_file.read()` загружал весь файл в RAM целиком, а синхронная функция `write_bytes()` блокировала Event Loop.
* **Стало:** Реализована асинхронная потоковая запись порциями (чанками) по **1 МБ** с помощью библиотеки `aiofiles`. Теперь сервер может принимать файлы любого размера (например, видео 10 ГБ), используя при этом **не более 1 МБ оперативной памяти**, абсолютно не блокируя обслуживание других пользователей.

#### Б. Потоковый парсинг в Celery (Алгоритм скользящего окна)
* **Было:** Парсинг `.txt` и `.pdf` читал весь файл в память воркера целиком.
* **Стало:** 
  * Для текстовых файлов внедрено построчное ленивое чтение.
  * Для PDF-файлов написан **алгоритм скользящего окна (Sliding Window)**: файл читается чанками по 1 МБ с сохранением "перекрытия" на стыке чанков, что позволяет точно подсчитывать теги `/Type /Page` без загрузки гигабайтных PDF-файлов в оперативную память.

### ⚙ 4. Стабилизация фоновых задач (Celery + Asyncio)

При работе асинхронной SQLAlchemy внутри синхронного воркера Celery возникали критические ошибки конкурентности (`InterfaceError: another operation is in progress` и `RuntimeError: Task got Future attached to a different loop`).

**Как решено:**
1. **Изоляция пулов при Fork:** Использован сигнал `@worker_process_init.connect` для утилизации родительского пула соединений SQLAlchemy `engine.sync_engine.dispose()` при форке процессов Celery. Каждый дочерний воркер теперь работает в своем изолированном сетевом сокете.
2. **Синглтон Event Loop:** Написан безопасный менеджер асинхронного цикла событий `run_async`, который создает и поддерживает единый постоянный Event Loop для каждого процесса воркера, предотвращая конфликты петель событий при выполнении последовательных задач.

### 🧪 5. Модульное тестирование (Pytest)

Благодаря архитектуре с внедрением зависимостей (DI), мы получили возможность проводить честное модульное тестирование роутеров без необходимости поднимать реальную БД или Redis.

* Написаны асинхронные тесты API с использованием `httpx.AsyncClient` и `pytest-asyncio`.
* С помощью `app.dependency_overrides` в тестах реальный сервис подменяется легковесным моком (`MockFileService`).
* С помощью `@patch` перехвачены вызовы Celery-задач.
* Все тесты и асинхронные фикстуры настроены в едином файле конфигурации `pyproject.toml`.

# Запуск тестов из папки backend
